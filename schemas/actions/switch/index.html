Switch

Routes the workflow down different branches based on the value of a specified input field. Define the routing rules in the 'condition' fields of the outgoing connections.

Overview

The Switch action provides multi-way branching for your workflows, similar to a switch statement in programming. Unlike the Conditional action which handles simple if/then logic, Switch excels at routing data down multiple different paths based on specific field values. Perfect for status-based routing, category handling, priority levels, and any scenario where you need to direct workflow execution based on discrete values.

🔀 Multi-way Routing

Route workflows down multiple different paths based on field values or conditions.

📊 Data Preservation

Passes input data through unchanged to the selected branch, maintaining data integrity.

⚡ Zero Configuration

No setup required - all routing logic is defined on the outgoing connections.

Switch Routing Examples

See how Switch routes data down different paths based on field values.

Ticket Status Routing

Route support tickets based on their status

Input Data:

{
  "ticket": {
    "id": "tkt-456",
    "status": "pending_approval",
    "priority": "high"
  },
  "user": "user-abc"
}

Connection Conditions:

ticket.status == "new" → new_ticket_handler
ticket.status == "pending_approval" → approval_workflow
ticket.status == "in_progress" → progress_tracker
ticket.status == "resolved" → resolution_handler
ticket.status == "closed" → archive_ticket

Result: Data flows to approval_workflow because status is "pending_approval"

Common Condition Patterns

Typical patterns for defining switch conditions on connections.

Exact Value Matching

// String values
status == "pending"
category == "electronics"
type == "premium"

// Numeric values
priority == 1
level == 5

Multiple Value Matching

// OR conditions
status == "new" || status == "pending"

// Array inclusion
["admin", "manager"].includes(role)

// Range matching
priority >= 1 && priority <= 3

Nested Field Access

// Object properties
user.subscription.tier == "premium"
order.shipping.method == "express"
product.category.name == "books"

Default/Fallback Routes

// Catch-all condition
true

// Null/undefined handling
status == null || status == undefined

// Default for unmatched values
!["new", "pending", "approved"].includes(status)

Common Use Cases

📋 Status-Based Routing

Route items based on their current status: new, pending, approved, rejected, completed.

👥 Role-Based Access

Direct users to different workflows based on their role or permission level.

🏷️ Category Processing

Handle different product categories, content types, or data classifications differently.

⚡ Priority Handling

Route high-priority items to express processing and normal items to standard queues.

🌍 Geographic Routing

Direct requests to region-specific processing based on location or timezone.

💳 Payment Processing

Route to different payment processors based on payment method or currency.

Best Practices

🎯 Explicit Conditions

Use explicit value matching for clarity:
• status == "approved" instead of status
• priority == 1 instead of priority

🛡️ Default Routes

Always include a default/fallback route:
• Use true as the last condition
• Handle unexpected or null values
• Log unmatched cases for debugging

📝 Descriptive Connections

Name your connections clearly:
• "approved_orders" instead of "connection1"
• "high_priority_queue" instead of "queue_a"
• "admin_workflow" instead of "admin"

🔍 Condition Order

Order conditions by specificity:
• Most specific conditions first
• General conditions in the middle
• Default/fallback condition last
